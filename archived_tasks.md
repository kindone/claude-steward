## Fixed Bugs

- [x] **Orphaned worker processes after abrupt e2e test exit** ‚Äî `worker.e2e.test.ts` spawned `tsx src/worker/main.ts` but `afterAll` cleanup didn't run on abrupt exits (vitest timeout, SIGTERM), leaving orphan processes that spiked EC2 load to 50+. Fix: registered `process.on('exit'/'SIGTERM'/'SIGINT')` signal handlers in `beforeAll` to guarantee `workerProc.kill('SIGTERM')` even without `afterAll`. Mitigation: `ps aux | grep tsx | grep -v grep | awk '{print $2}' | xargs kill`.
- [x] **Graceful context limit / error UX** ‚Äî added `context_limit` error code detected from Claude CLI result text; friendly yellow banners per error type (`context_limit`, `session_expired`, `process_error`, `http_error`) replacing the raw CLI error string; partial streamed content now preserved above the error banner rather than overwritten.
- [x] **Push fires on active sender** ‚Äî `notifyWatchers` only tracks `watchSession` connections, not active `sendMessage` SSEs, so `notified === 0` was always true during live sends and push fired unconditionally. Fixed by also checking `res.writableEnded` ‚Äî push now only fires when the sending client has disconnected.
- [x] **VapidPkHashMismatch subscription not cleaned up** ‚Äî Apple APNs 400 `VapidPkHashMismatch` (stale subscription from a previous VAPID key rotation) was logged but the subscription was never deleted, causing every send to fail permanently. Now treated as permanent like 410/404 and auto-deleted.
- [x] **Last project/session not restored on reload** ‚Äî the persist effect was firing with `activeSessionId=null` during loading transitions (Phase 2 sets it null before the async `listSessions` resolves), overwriting the stored session ID before it could be restored. Fixed by guarding on both `activeProjectId && activeSessionId` before saving.
- [x] **Interrupted session handling** ‚Äî errors (token limit, process crash, etc.) are now persisted to the DB as assistant messages with `is_error=1` and `error_code`. `onError` in `chat.ts` inserts the error message before calling `notifyWatchers`, so a reloaded tab re-fetches and renders the error bubble correctly. `dbMessageToLocal` in `ChatWindow` maps `is_error` ‚Üí `errorCode` for all load paths (initial, watch, pagination).
- [x] **Push notification hardening** ‚Äî VAPID credentials now initialised once per process (lazy `vapidInitialised` flag) instead of on every send. Transient failures (5xx, network errors) are retried once after 1 s with structured logging; stale 410/404 subscriptions continue to be auto-deleted.
- [x] **XSS in MessageBubble** ‚Äî `marked.parse()` output was inserted via `dangerouslySetInnerHTML` with no sanitization; crafted Claude output could execute arbitrary scripts. Fixed by adding `dompurify` to the client and wrapping the `marked.parse()` result with `DOMPurify.sanitize()` before insertion.
- [x] **Spinner stuck after page reload + error** ‚Äî `onError` in `chat.ts` did not call `notifyWatchers()`, so any tab that reloaded mid-stream and parked on the watch SSE endpoint would never receive a `done` event and its spinner would hang forever. Fixed by calling `notifyWatchers(sessionId)` in the `onError` callback.
- [x] **`onDone` fires after `onError` (double callback)** ‚Äî `doneFired` in `api.ts` was only set on `event: done`, so when the stream closed after an `event: error` the fallback `if (!doneFired) handlers.onDone()` would still fire. Added `errorFired` flag to guard the fallback.

- [x] **WebAuthn RP ID mismatch on new devices** ‚Äî `APP_DOMAIN` was hardcoded as `steward.example.com` in `ecosystem.dev.config.cjs`, overriding the real value in `.env`; removed the hardcoded value so `dotenv` loads it correctly.
- [x] **VAPID keys not loading (503 on push subscribe)** ‚Äî three-layered fix: (1) removed empty VAPID key entries from PM2 ecosystem configs that `dotenv` refused to override; (2) moved `process.env` reads from module init to inside functions in `pushNotifications.ts` to avoid ESM hoisting order issues; (3) used `--update-env` on `pm2 restart` to apply new `.env` values.
- [x] **`sw.js` JSON parse crash from DevTools test push** ‚Äî `event.data?.json()` threw on the plain-string payload sent by Chrome DevTools' Push button; wrapped in `try/catch` with `event.data?.text()` fallback.
- [x] **Conversation scroll jumps to top on iOS notification tap** ‚Äî initial load used `scrollIntoView({ behavior: 'smooth' })` which iOS interrupted when the tab became active; fixed by using `'instant'` on first load (switching to `'smooth'` after), and a `scrollBehaviorRef = 'none'` + `scrollHeight` anchor when prepending older messages.
- [x] **Session state ambiguous after mid-stream tab close** ‚Äî when both server and worker restarted, recovery marked jobs as `interrupted` via `markStaleStreamingMessages()` but never called `notifyWatchers`, leaving any open `watchSession` connections spinning indefinitely. Fixed in `recovery.ts`: `settle()` now calls `notifyWatchers` for all in-flight sessions after `markStaleStreamingMessages()`; already-resolved sessions have empty watcher sets so the extra calls are no-ops.
- [x] **Red error bubble after tab-close-and-return mid-stream** ‚Äî after `watchSession` fired `done` and the async `getMessages` resolved, a spurious `onerror` from the server closing the TCP connection after sending `event: done` fired last and stamped the message as error. Fixed in `watchSession` (`api.ts`) by adding a `doneFired` guard: `onerror` is a no-op once `done` has already fired.
- [x] **Scroll stutters during streaming response** ‚Äî repeated `scrollIntoView({ behavior: 'smooth' })` calls on each streaming chunk interrupted each other, causing jitter. Fixed in `ChatWindow.tsx`: streaming deltas now use direct `scrollTop = scrollHeight` (no animation) and only fire when the user is within 100px of the bottom (respects deliberate scroll-up).
- [x] **VAPID private key committed to git history** ‚Äî hardcoded keys and personal domain/email removed from `ecosystem.dev.config.cjs`; new VAPID keys generated and stored only in `.env`.
- [x] **Push notification tap didn't navigate to session on mobile** ‚Äî three layered bugs: (1) SW focused the existing tab but never changed its URL ‚Äî fixed by using `postMessage({ type: 'switchSession' })` instead of unreliable `client.navigate()`; (2) push URL lacked `project_id` so the app couldn't select the right project on fresh open ‚Äî fixed by including `&project=<id>` in the scheduler push payload; (3) `pendingSessionIdRef` was consumed in the null-project sessions-effect run (all-sessions placeholder fetch) before the real-project run could use it ‚Äî fixed by only clearing the ref when `activeProjectId !== null`
- [x] **nginx dev domain pointing to wrong upstream** ‚Äî `proxy_pass` in `steward-dev` was changed to `:3002` during a build-watch experiment, breaking HMR; reverted to `:5173` (Vite dev server) with WebSocket upgrade headers already in place via `ws-map.conf`.
- [x] **Mobile scroll stutter on touch-end** ‚Äî `setIsAtBottom(true)` in the scroll-settle debounce triggered a React re-render that interrupted the browser's native momentum deceleration near the bottom of the chat, causing visible stutter. Fixed by converting `isAtBottom` from `useState` to a ref; scroll-to-bottom button visibility is now toggled via direct DOM class manipulation (`scrollBtnRef.current.classList`), eliminating all re-renders from the scroll path.
- [x] **Initial scroll doesn't reach bottom on sessions with async content** ‚Äî `scrollTop = 1e9` on initial load fired before mermaid/images/KaTeX finished rendering, leaving the view short of the true bottom. Two bugs: (1) the programmatic `scrollTop` assignment fired a scroll event that set `userIsScrollingRef = true` after just 8ms, killing the rAF catchup loop; (2) no catchup mechanism existed to re-snap after late layout shifts. Fixed by setting `skipNextScrollRef` before every programmatic scroll and adding a 5-second rAF loop that re-snaps whenever `scrollHeight` grows. Diagnosed remotely on iPhone via `/api/eval` ‚Äî storing debug data on `window.__scrollDebug` and reading it server-side.
- [x] **Push notification tap doesn't navigate to session on iOS** ‚Äî iOS Safari doesn't fire the SW `notificationclick` event, so `postMessage({ type: 'switchSession' })` never reaches the page. Three failed approaches: (1) SW `postMessage` from `notificationclick` ‚Äî iOS doesn't fire it; (2) `BroadcastChannel` from SW to page ‚Äî doesn't work on iOS; (3) SSE `pushTarget` broadcast ‚Äî iOS kills the SSE connection when backgrounded. Final working approach: server stores last push target in memory (`setLastPushTarget`); `GET /api/push/last-target` returns and clears it; page polls on `visibilitychange` and navigates if target is < 60s old. Also fixed: chat route and manual schedule run push URLs missing `&project=` param; SW `notificationclick` now extracts and passes `projectId`; App.tsx `switchSession` handler now switches project via `setActiveProjectId` + `pendingSessionIdRef` instead of full page reload.
