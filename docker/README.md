# Docker — Claude Steward

## Why a long-lived token

The container authenticates as your Claude account via a long-lived OAuth
token generated by `claude setup-token`. This is an **independent credential**
from the rotating `accessToken` / `refreshToken` pair in
`~/.claude/.credentials.json` that your host CLI uses day-to-day.

Why this matters: refresh tokens rotate on use. The original strategy of
copying `~/.claude/.credentials.json` into the container failed because the
container and the host then shared the same rotating credential — whichever
refreshed first invalidated the other, and the host CLI got logged out.

`setup-token` sidesteps that entirely: it issues a separate, non-rotating
credential for the same account. The container uses it; your host keeps its
own rotating creds; the two never interfere. No separate Claude account
required.

## Prerequisites

### 1. Generate the long-lived token

Run `claude setup-token` against a scratch config dir so your prod
`~/.claude/` is never touched during login:

```bash
CLAUDE_CONFIG_DIR=$HOME/.claude-test claude setup-token
```

A browser window opens. Sign in with your normal Claude account — the token
this generates is independent of the one your daily `~/.claude/` uses, so
there's no conflict. The CLI prints a long-lived token — copy it.

The `CLAUDE_CONFIG_DIR` override is a defensive measure: it ensures the
`setup-token` flow can't accidentally modify your prod `~/.claude/` on disk.
Once the token is printed, the scratch `~/.claude-test/` can be deleted.

### 2. Store the token in `.env`

At the project root:

```bash
echo 'STEWARD_TEST_OAUTH_TOKEN=<paste-token-here>' >> .env

# Optional: set a fixed API key so you can log into the steward UI without
# going through the passkey flow each test run.
echo 'API_KEY=<some-secret>' >> .env
```

`.env` is already gitignored.

### 3. (Optional) `.env.container` for app-level vars

If the container needs `APP_DOMAIN`, `VAPID_PUBLIC_KEY`, etc.:

```bash
cp .env docker/.env.container
# Then uncomment the .env volume line in docker-compose.yml
```

## Running

```bash
# Build image and start
docker compose up --build

# Subsequent starts (cached image)
docker compose up -d

# Stop (keeps DB volume intact)
docker compose down

# Stop + wipe DB (clean slate for testing)
docker compose down -v
```

App: `http://localhost:13001` — Safe fallback: `http://localhost:13003`
(mapped from container ports 3001/3003).

## How auth works

The container reads `CLAUDE_CODE_OAUTH_TOKEN` from its environment (passed in
by `docker-compose.yml` from `STEWARD_TEST_OAUTH_TOKEN` in your `.env`). The
Claude CLI treats this as its primary auth credential — no credentials file
is read or written for auth purposes.

The entrypoint also:

- `unset ANTHROPIC_API_KEY` — prevents the CLI from routing through API
  credits if the var leaks in via the environment.
- Bootstraps an empty `/root/.claude.json` on first run so the server's
  `syncClaudeSettings()` can register the `steward-schedules` and
  `steward-artifacts` MCP servers for the in-container Claude CLI.

### Token lifecycle

| Phase | Action |
|---|---|
| Initial setup | `claude setup-token` → copy into `.env` as `STEWARD_TEST_OAUTH_TOKEN` |
| Normal operation | Nothing — the long-lived token doesn't rotate |
| Token revoked / expired | Re-run `setup-token`, update `.env`, `docker compose restart` |

Because the token doesn't rotate, running the container **cannot** invalidate
any other session — including your own prod `~/.claude/`.

## Ports

| Host port | Container port | Service |
|---|---|---|
| 13001 | 3001 | Main server (HTTP API + static files) |
| 13003 | 3003 | Safe-mode fallback |

Mini-app ports (4001–4010) are not exposed by default — add them to
`docker-compose.yml` if needed.
